<AIExplain title="Real workflow">
Write `.ASM` in MPLAB → MPASM produces `.HEX` → load `.HEX` in Proteus (simulation) or a programmer (physical board) → compare behavior with what you expect.

## What MPLAB is (and what it is not)

Reports and labs mix three names. Keep them separate:

| Tool | What it is | Course role |
| --- | --- | --- |
| **MPLAB v8 / X** | **IDE** — development environment | Edit `.ASM`, create project, run build, program with PICkit |
| **MPASM** | **Assembler** | Translates `.ASM` → `.HEX` (program + [configuration word](/en/fundamentos/config-bits/#what-is-the-configuration-word)) |
| **Proteus ISIS** | **Simulator** | Runs `.HEX` in a virtual circuit — **does not compile** |
| **PICkit / programmer** | **Burner** | Writes `.HEX` to real PIC Flash |

**MPLAB is not the assembler.** MPLAB is the **workspace**; when you click **Build**, MPLAB calls **MPASM** for you.

### Step 1 — Write the `.ASM` in MPLAB

Create a **PIC18F4550** project and edit source using the [sandwich](/en/fundamentos/ensamblador/#anatomy-of-an-mpasm-file--sandwich-layout) layout: `LIST`, `#include`, `CONFIG`, `ORG`, subroutines, `programa:` at the bottom.

The file on disk is plain text (`.ASM` / `.asm`). MPLAB **saves and highlights** it — there is no `.HEX` yet.

### Step 2 — Build with MPASM

**Project → Build All** invokes MPASM. If there are no errors, you get the **`.HEX`**:

```
  your_program.ASM
       ↓  MPASM (from MPLAB)
  your_program.HEX   ← Flash program + config bits
  your_program.COD   ← listing (optional, debug)
```

If `END` is missing, a symbol lacks `#include`, or a label is wrong, the build **fails** — **do not simulate** until you fix the **Output** window.

### Step 3 — Simulate in Proteus

Wire the [minimum system](/en/fundamentos/config-bits/#minimum-system--hardware-that-goes-with-config-bits) + LED, load the **same `.HEX`** into the virtual PIC, and press **Play**. Proteus **interprets** `.HEX` bytes; it does not read your `.ASM` directly.

### Step 4 — Physical board

With the **same `.HEX`** (rebuilt if you changed code), connect PICkit, **Program** in MPLAB, and validate the LED or peripheral on the breadboard.

<AIExplain title="Course golden rule">
**Build without errors → simulate in Proteus → breadboard.** Not the other way around. Proteus never replaces MPASM.
</AIExplain>

## What the `.HEX` file contains

The `.HEX` is the **bridge** between software and hardware. It is not readable `.ASM` — Intel HEX format with **two conceptual blocks**:

| `.HEX` content | Source in your `.ASM` | Destination |
| --- | --- | --- |
| **Program** (instructions) | `ORG`, `GOTO`, `MOVLW`, `BSF`… | Executable Flash |
| **Configuration word** | `CONFIG` directives | Config memory (startup) |

One file feeds Proteus **and** the programmer: both need program **and** config bits matching the **20 MHz** crystal and course template.

## Full chain — from text to the PIC

```
  MPLAB editor (.ASM)
       ↓  Build / assemble
  MPASM → .HEX file (+ optional .cod)
       ↓
  ┌────────────────┴────────────────┐
  ↓                                 ↓
Proteus (simulation)          PICkit / programmer
  ↓                                 ↓
Virtual LED, logic pins       UNEXPO physical board
```

| Stage | Tool | Input | Output |
| --- | --- | --- | --- |
| Write | MPLAB v8 / X | Your source code | `.ASM` |
| Assemble | **MPASM** | `.ASM` | **`.HEX`** (program + config bits) |
| Simulate | **Proteus ISIS** | `.HEX` + schematic | Virtual behavior |
| Program | PICkit 2/3, etc. | `.HEX` | PIC18F4550 Flash |

## Full walkthrough — minimal example (First LED)

Suppose you turn on RB0. The full flow:

**1. Source in MPLAB** — minimal fragment (see [Assembly — LED](/en/fundamentos/ensamblador/#full-example--led-on-rb0)):

```asm
        LIST    P=18F4550
        #include <P18F4550.INC>
        CONFIG  FOSC = HS, WDT = OFF, LVP = OFF, PBADEN = OFF

        ORG     0x0000
        GOTO    programa         ; jump to main

programa:
        BCF     TRISB, 0
        BSF     LATB, 0
loop:
        GOTO    loop               ; jump → destination loop: (above)

        END
```

**2. Build in MPLAB** → `project.hex` appears with no errors in **Output**.

**3. Proteus** — PIC18F4550 + 20 MHz crystal + LED on RB0 → **Program File** = that `.HEX` → **Clock Frequency** = **20 MHz** → **Play**.

**4. Board** — same `.HEX`, PICkit, **Program** → LED on if wiring and [config bits](/en/fundamentos/config-bits/#summary--course-template-20-mhz) match.

If you change one `.ASM` line, **go back to step 2** before Proteus or the board. An old `.HEX` is the #1 cause of “simulation does not show my change”.

## MPASM — what happens when you build

MPASM reads your [assembly file](/en/fundamentos/ensamblador/#anatomy-of-an-mpasm-file) and:

1. **Translates** mnemonics (`MOVLW`, `BSF`…) into machine code.
2. **Resolves** labels, `#include` symbols, and `CONFIG` directives.
3. **Generates** the `.HEX` for programmer or Proteus.
4. **Reports errors** if `END` is missing, labels duplicate, or symbols are undefined.

### Files you see after a build

| File | Purpose |
| --- | --- |
| **`.HEX`** | Load into Proteus or programmer — **the important one** |
| **`.COD`** | Listing with addresses and machine code (debug) |
| **`.LST`** | Source mixed with listing (if enabled) |
| **`.ERR`** | Assembly errors (if build fails) |

If the build fails, **there is no useful `.HEX`** — fix errors in the output window before simulating.

### MPASM errors you will see in MPLAB

| Output message | Common cause | Review |
| --- | --- | --- |
| `Symbol not defined` | Missing `#include <P18F4550.INC>` or typo (`TRISB`) | [Assembly — directives](/en/fundamentos/ensamblador/#directives-you-must-know) |
| `Label required` | Label without `:` (`programa` vs `programa:`) | [Labels](/en/fundamentos/ensamblador/#labels-and-comments) |
| `END directive required` | Missing `END` at end | [Sandwich layout](/en/fundamentos/ensamblador/#anatomy-of-an-mpasm-file--sandwich-layout) |
| `Bank select error` | Variable in another bank without `BSR` | [Registers — banks](/en/fundamentos/registros/#data-memory-map) |
| `Argument out of range` | Literal outside 0–255 | [Literals](/en/fundamentos/ensamblador/#mpasm-notations-for-literals) |

## MPLAB v8 project — what the wizard creates

The **Project Wizard** does not write code for you; it organizes files and tells MPASM which MCU and tool to use:

| Project element | Function |
| --- | --- |
| **PIC18F4550 device** | MPASM generates correct opcodes for that chip |
| **MPASM toolchain** | Not XC8 if the course is pure assembly |
| **Added `.asm` file** | Source you build |
| **Project folder** | Where `.HEX` appears after **Build All** |

After building, find the `.HEX` in the project folder (sometimes `Debug/`, sometimes root). That path is what you load in Proteus → **Program File**.

## MPLAB v8 — main course IDE

MPLAB v8 is Microchip's classic IDE. UNEXPO labs often use it because it fits **MPASM** and pure `.ASM` projects well.

		### When to use it

		MPLAB X is the modern IDE (NetBeans-based). Microchip promotes it for new **XC8** projects, but it can assemble `.ASM` with the right toolchain.

		### Summary steps

		1. **File → New Project** → Standalone Project
		2. Device: **PIC18F4550**
		3. Pick a toolchain compatible with assembly
		4. Add your `.asm` to the project
		5. **Production → Build Main Project**
		6. Check **Output** and locate the `.hex`

		If your instructor requires MPLAB v8, **do not mix** v8 and X projects in the same folder — paths and settings differ.
	</TabItem>
	<TabItem label="Proteus">
		### Build the circuit

		1. Place **PIC18F4550** on the schematic
		2. Wire [minimum system](/en/fundamentos/config-bits/#minimum-system--hardware-that-goes-with-config-bits): **20 MHz** crystal, capacitors, MCLR pull-up, VDD/VSS, decoupling
		3. Add LED + resistor on **RB0** (or your lab pin)
		4. Check common ground and LED polarity

		### Load the .HEX

		1. Double-click the PIC → **Program File** tab
		2. Select the **`.hex`** from MPASM
		3. Set **Clock Frequency** to **20 MHz** (matches `FOSC = HS` and crystal)
		4. OK and save schematic

		### Run simulation

		1. **Play** (Run simulation)
		2. Watch virtual LED and **RB0** levels
		3. Use **Debug** to inspect registers if your version allows
		4. If it does not start: check loaded `.HEX`, clock frequency, and [config bits](/en/fundamentos/config-bits/)

		Proteus **does not assemble** — it only runs the `.HEX` you built in MPLAB.
	</TabItem>
	<TabItem label="Physical board">
		### Before programming

		1. Stable **5 V** supply and decoupling cap
		2. 20 MHz crystal + caps + MCLR 10 kΩ ([minimum system](/en/fundamentos/config-bits/))
		3. Check LED wiring (220–330 Ω resistor)
		4. No shorts on the breadboard

		### Program the MCU

		1. Connect **PICkit 2/3** (or lab programmer) to ICSP
		2. In MPLAB: **Programmer → Select Programmer**
		3. **Program** with the same `.HEX` used in Proteus
		4. Confirm *Programming/Verify complete*

		### Validate

		1. LED on/off per your code
		2. Multimeter on output pins if something fails
		3. Compare with simulation — Proteus OK but board not → suspect **hardware** (wiring, supply, wrong config bits)
	</TabItem>

## MPLAB vs Proteus vs board — who does what

| Task | MPLAB + MPASM | Proteus | Physical board |
| --- | --- | --- | --- |
| Write code | ✅ | ❌ | ❌ |
| Build `.HEX` | ✅ | ❌ | ❌ |
| Simulate without hardware | ❌ | ✅ | ❌ |
| Real crystal timing | ❌ | Approximate | ✅ |
| Noise, contacts, bounce | ❌ | Limited | ✅ |
| Program PIC Flash | With programmer | Simulates only | ✅ |

Course rule: **build without errors first**, **then simulate in Proteus**, **then breadboard**.

## Cross-tool consistency — checklist

Before saying “it does not work”, verify **everything matches**:

- [ ] Device: **PIC18F4550** in MPLAB, Proteus, and board
- [ ] `CONFIG FOSC = HS` and **20 MHz** crystal in schematic and Proteus
- [ ] Same **`.HEX`** in Proteus and programmer (rebuild after every change)
- [ ] `PBADEN = OFF`, `LVP = OFF`, `WDT = OFF` (course template)
- [ ] LED pin matches code (`RB0` → `BCF TRISB,0` and `BSF LATB,0`)
- [ ] Rebuilt after editing `.ASM`

2. **Build** in MPLAB → get `.HEX`

## Common mistakes

- **Proteus without loaded `.HEX`**: PIC simulates empty memory or old program.
- **Proteus clock not 20 MHz**: timing and some peripherals won't match the board.
- **Mixing MPLAB X and v8 projects**: conflicting paths and toolchains.
- **Simulation OK, board not**: check supply, MCLR, crystal, PIC orientation, LVP/PBADEN.
- **Board OK, simulation not**: outdated `.HEX` or wrong Proteus frequency.
- **Treating Proteus as compiler**: Proteus **does not replace** MPASM — it only runs `.HEX`.

## Exam-style questions

**What is the difference between MPLAB and MPASM?**

**MPLAB** is the **environment** (editor, project, programmer). **MPASM** is the **assembler** that translates `.ASM` → `.HEX` when you Build.

**What file does MPASM produce on a successful build?**

A **`.HEX`** file with program and config bits for programming or simulation.

**What is Proteus used for in the course?**

To **simulate** circuit and program **before** the physical board, loading the `.HEX` from MPLAB.

**Does MPLAB assemble directly or use another tool?**

MPLAB invokes **MPASM** (for pure assembly projects in the course).

**Why must Proteus clock frequency match the crystal?**

Execution time and peripherals depend on **Fosc**; with `FOSC = HS` and 20 MHz crystal, Proteus should use **20 MHz**.
